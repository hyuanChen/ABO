# ABO — AGENTS.md

本文件写给会继续在 ABO 上开发的代理、协作者和未来的自己。

它不是产品介绍，而是开发约束。进入仓库后，先读这个文件，再动代码。

---

## 0. 先理解这不是一个普通桌面应用

ABO 的核心不是“多做几个页面”，而是把用户的长期输入整理成一个本地、持续、可沉淀的系统。

当前产品主要围绕三条主链运转：

1. `研究链`
   `论文发现 -> 今日情报 -> 阅读筛选 -> 文献库 / Wiki -> 助手生成判断与 idea`

2. `注意力链`
   `平台输入 -> 热点归类 -> 偏好记录 -> 情报库 -> 日报 / 长期注意力画像`

3. `成长链`
   `个人记录 -> 状态总结 -> 规律识别 -> 建议输出 -> 虚拟陪伴 / 自我建模`

任何新功能，先回答两个问题：

1. 它属于哪一条链？
2. 它最后沉淀到哪里？

如果这两个问题答不清，通常说明功能还没有真正嵌进 ABO，而只是一个孤立工具。

---

## 1. 最高优先级规则：定时任务必须复用主动工具链

- 任何“每日情报 / 定时任务 / feed 监控”功能，必须复用对应“主动工具”的前端组件、后端抓取逻辑、卡片结构、保存路径、入库逻辑、过滤逻辑和配置结构。
- 不允许为了定时任务另外写一套平行实现；定时任务本质上只是“在定时触发时执行一次主动工具已有链路”。
- 如果主动工具已经有：
  - 卡片组件
  - crawl / fetch / process 逻辑
  - 保存到文献库 / Wiki / Vault 的逻辑
  - 过滤配置、页数、条数、时间窗等参数
  - 调试入口
  那么 feed / 定时任务必须直接复用这些实现，优先抽公共函数或公共组件，不能在 feed 里复制一份简化版或残缺版。
- 对论文、Follow Up、小红书、Bilibili 都适用这条规则。

开始实现任何 feed / 定时任务需求前，先检查：

1. 对应主动工具的现有前端组件是否能直接复用
2. 对应主动工具的后端 fetch / process / save 是否能直接复用
3. 当前改动是否只是“调度 + 配置透传 + 去重 / 展示适配”

如果发现 feed 和主动工具出现不一致，默认判断为 bug，优先修成“主动工具单一事实来源”，而不是继续补 feed 专用分支。

---

## 2. 单一事实来源规则

在 ABO 里，功能的事实来源有明确层级：

1. `当前可用的主动工具链`
   最优先。手动工具页已经能跑通的链路，才值得被调度化、摘要化、模块化。

2. `本地保存结果`
   包括 `Vault / Literature / Wiki / ~/.abo/* / SQLite cards.db`。这些是长期状态，不是临时 UI 状态。

3. `后端配置与模块配置`
   配置必须回写到统一配置位置，不能一处工具页一套、一处调度一套。

4. `前端展示状态`
   前端负责读、触发、反馈，不应该变成业务真相的唯一来源。

如果一个需求让你想“直接在前端多存一点状态先凑合”，先停一下。ABO 大多数长期问题最后都会回到本地存储和统一配置上。

---

## 3. 当前项目的真实状态

不要把仓库想成“干净、统一、完全收口”的状态。当前有几个事实需要接受：

1. `代码比文档更新`
   `README.md`、`docs/LLM-wiki.md`、`docs/superpowers/*` 很有用，但其中一部分也可能落后于现状。碰到冲突时，以当前代码为准。
4. `命名和 AI 提供者有历史包袱`
   仓库里仍能看到 Claude、Codex、旧聊天桥、旧画布等多阶段遗留。现在开发时不要急着大面积“统一命名”，除非用户明确要求做一次整理。
5. `有些页面是产品方向清晰、实现还在演化`
   例如手记、健康管理、部分分析视图已经有 UI 面，但不一定都接上完整真实数据。改动前先分清楚它是“可用链路”还是“占位 / 演示 / 半成品”。

## 4. 开发前的最小阅读顺序

开始一项改动前，建议按这个顺序看：

1. `README.md`
2. `docs/abo-user-guide.md`
4. `AGENTS.md`
5. 相关模块的实现代码

如果你只看旧文档不看代码，很容易沿着已经废弃的路径继续开发。

补充说明：

- 当前真正影响 Wiki 维护方式的文档，是 `docs/LLM-wiki.md`。

## 4.5 Git 工作方式

1. 默认直接在仓库当前主工作树开发，不要额外新开 `git worktree` 作为日常开发路径。
2. 默认直接在 `main` 分支上继续推进当前开发，除非用户明确要求切分支或单独做隔离实验。
3. 如果仓库已经存在其他 worktree，把它们视为历史产物，不要把新的改动继续落到那些 worktree 里。
4. 提交前先确认当前目录就是用户正在使用的主仓库目录，再执行 `git add` 和 `git commit`。

## 5. 新需求应该怎么落地

### 5.1 先给需求归类

先判断它属于哪种改动：

- `主动工具增强`
- `定时模块 / feed 能力`
- `库 / Wiki / 助手沉淀`
- `角色主页 / 生活管理 / 成长面板`
- `纯展示页 / 交互优化`

ABO 里最常见的错误，就是用户要的是“整条链更顺”，结果实现成了一个孤立小按钮。

### 5.2 先找到它挂在哪个现有入口

优先挂到现有侧边栏语义里：

- `角色主页`
- `助手`
- `今日情报`
- `数据洞察`
- `情报库`
- `文献库`
- `Wiki`
- `手记`
- `模块管理`
- `小红书工具`
- `哔哩哔哩工具`
- `论文追踪`
- `健康管理`

不要轻易再发明一个新的一级入口，除非用户明确要重构 IA。

### 5.3 手动链先跑通，再自动化

对采集类、监控类、入库类功能，顺序必须是：

1. 工具页手动执行
2. 结果预览正确
3. 保存路径正确
4. 进入情报库 / 文献库 / Wiki 正确
5. 再接到模块管理和定时任务
6. 最后再接到今日情报、助手和数据洞察

跳过前几步，直接做自动化，后面大概率会返工。

### 5.4 优先做端到端薄切片

ABO 更适合做“小而完整”的新链路，而不是只做一层 UI 或只做一个底层工具函数。

一个合格的新功能切片，至少应该能回答：

- 用户从哪个入口触发
- 看到了什么反馈
- 内容最后保存到哪里
- 后续能不能在 Feed / 库 / Wiki / 助手里继续被利用

### 5.5 Wiki 相关需求，优先按 LLM Wiki 心智实现

ABO 里的 Wiki，不应按普通“手写笔记目录”理解，而应按 `LLM Wiki` 模式理解：

`raw sources`
-> `LLM 持续提取、归并、交叉引用`
-> `形成持久 Markdown wiki`
-> `用户通过 Obsidian / UI / 助手继续查询、修改和扩写`

这意味着：

1. Wiki 不是单纯的展示层
2. Wiki 也不是 query-time RAG 的一次性回答缓存
3. 好的总结、分析、比较、关系发现，都应该考虑是否写回 Wiki
4. 做 Wiki 相关需求时，先判断自己在做的是：
   - `ingest`
   - `query`
   - `lint`

如果一个改动只能让聊天回答更长，但不会让 Wiki 更完整，通常还没有真正长进 ABO 的知识层。

## 6. 前端约束

1. 优先复用现有页面骨架和 `Layout` 组件。
2. 新页面先接入 `NavSidebar -> store.ActiveTab -> MainContent` 这一套现有导航结构。
3. 全局状态优先进 `src/core/store.ts`，不要到处发散局部全局状态。
4. API 调用统一走 `src/core/api.ts`。
5. 提示和错误反馈统一走 Toast 或页面内明确状态，不要静默失败。
6. 配置面板遵守“渐进式披露”：基础默认清晰，高级选项折叠。
7. 不要为了快而在不同页面复制一份样式相近但逻辑不同的组件。
8. 先查是否已有对应卡片、面板或表单模式，再决定是否新建。

---

## 7. 后端约束

1. 新模块优先走 `abo/sdk/base.Module` 这条标准链。
2. 模块的 `fetch` 和 `process` 分层要清楚，避免把所有事塞到一个函数里。
3. 本地文件输出要保持稳定目录约定，不要频繁改路径命名。
4. 结果一旦有“长期使用价值”，就要考虑如何进入：
   - `cards.db`
   - `Vault`
   - `Literature`
   - `Wiki`
5. 对已有抓取逻辑，优先抽公共函数，不要新起平行路由。
6. 配置应尽量进入统一配置结构，而不是藏在脚本参数或临时本地状态里。

---

## 8. 文档与材料放置规则

1. `README.md`
   放外部整体介绍和核心能力。
2. `docs/`
   放当前还会被继续使用的用户指南、开发指南、专项设计。
3. `docs/LLM-wiki.md`
   放当前 Wiki 维护模式与知识层心智，是现行的 Wiki 方向入口。
4. `docs/superpowers/plans/`
   放阶段性实现计划、分步设计、未完全收敛的方案。

## 9. 当前最容易踩的坑

1. `不要重新创建 AGENT.md`
   当前 worktree 显示 `AGENT.md` 已被删除。除非用户明确要求，否则不要恢复。

2. `不要把调度功能做成主动工具的平行实现`
   这条已经是最高优先级规则，再强调一次。

3. `不要只做 UI 却不接保存链`
   ABO 很多功能的价值不在“看起来能点”，而在“数据最终沉到哪里，后面还能不能继续用”。

4. `不要忽视本地路径和目录约定`
   情报库、文献库、Wiki 路径错了，功能表面上能跑，实际对用户就是坏的。

5. `不要把旧设计稿当成现状`
   画布、生成稿和历史方案很多，只能拿来理解产品思路，不能直接代表今天代码里正在用什么。

6. `不要急着做大一统重构`
   这个仓库目前更适合“贴着现有主链补齐关键断点”，不适合一上来做命名、目录、提供者、UI 全面统一。

---

## 10. 完成一项改动前，最少要验证什么

### 采集 / 监控类

1. 手动工具页能跑
2. 结果预览合理
3. 保存路径正确
4. 今日情报能看到
5. 模块管理状态正常

### 库 / Wiki / 助手类

1. 结果能被写入本地
2. 页面能重新读取出来
3. 助手或后续链路能继续消费这些内容

### 角色 / 仪表板 / 生活类

1. 页面状态有真实来源，不全是硬编码
2. 变化能被保存或复现
3. 与情报 / 助手 / 手记之间至少有一条明确连接

---

## 11. 对代理最重要的一句话

在 ABO 里，好的开发不是“又做了一个功能”，而是：

`让已有输入更顺地流进系统，让已有内容更稳地沉下来，让已有知识更容易被再次调用。`

如果一个改动同时让 `入口更清楚`、`链路更一致`、`沉淀更稳定`，那通常就是好改动。
