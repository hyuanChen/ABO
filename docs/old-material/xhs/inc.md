# `xiaohongq-crawler-analysis.md` 与 `crawler_overall.md` 的实现差异

对比对象：

- `/Users/huanc/Desktop/local_viz/开源项目/xiaohongshu-skills/docs/xiaohongq-crawler-analysis.md`
- `/Users/huanc/Desktop/ABO/docs/xhs/crawler_overall.md`

目标是回答三件事：

1. 两份文档描述的实现到底有什么区别。
2. 第一份比第二份少了什么。
3. 第二份比第一份少了什么。

## 1. 先说总判断

两份文档都强调“合规采集，不讨论绕过平台风控”，但默认实现模型不一样。

### 第一份文档的默认实现模型

`xiaohongq-crawler-analysis.md` 站在当前 `xiaohongshu-skills` 仓库的真实实现上，主路径是：

`CLI -> 本地 bridge server -> Chrome 扩展 -> 当前真实浏览器 tab -> 页面主 world 取状态 / 必要时做交互`

它更像“当前代码是怎么跑的”的实现拆解文档。

### 第二份文档的默认实现模型

`crawler_overall.md` 的主路径是：

`输入归一化 -> 直接请求详情页 HTML -> 解析 __INITIAL_STATE__ -> 抽字段 -> 下载媒体 -> 落 Markdown / 本地库`

如果静态链路拿不到，再用真实浏览器 / CDP 做兜底。

它更像“ABO 侧长期采集系统应该怎么设计”的总体架构文档。

### 一句话区别

- 第一份偏“浏览器内执行方案”
- 第二份偏“内容采集同步系统方案”

## 2. 两份文档描述的实现区别是什么

## 2.1 默认抓取入口不同

### 第一份

默认入口是浏览器：

- 复用真实用户浏览器和登录态
- Python 通过本地 WebSocket 调扩展
- 扩展在页面主 world 执行 JS
- 优先直接读取页面已经渲染好的状态对象

浏览器不是兜底，而是主执行环境。

### 第二份

默认入口是后端详情页请求：

- 先把链接 / note id 归一化
- 直接请求详情页 HTML
- 从 HTML 中解析 `window.__INITIAL_STATE__`
- 只有静态链路不够时，才切到浏览器 / CDP

浏览器是兜底，不是主路径。

### 差异结论

这是两份文档最大的实现差异。

- 第一份强调“在浏览器里拿页面已有数据”
- 第二份强调“能不用浏览器就先不用浏览器”

## 2.2 关注层级不同

### 第一份更关注执行链路

重点是：

- `scripts/cli.py`
- `scripts/bridge_server.py`
- `scripts/xhs/bridge.py`
- `extension/background.js`
- `chrome.scripting.executeScript`
- 页面状态读取
- 页面交互、等待、滚动、点击

它回答的是：“采集动作在页面里怎么完成？”

### 第二份更关注系统链路

重点是：

- 输入归一化
- 列表发现与详情采集拆分
- 结构化字段抽取
- 媒体下载
- 本地状态存储
- 增量同步
- 调度、限速、熔断、冷却

它回答的是：“这套采集系统如何长期稳定运行？”

## 2.3 对页面状态读取的描述深度不同

### 第一份

第一份把页面状态读取讲得更具体，明确举了页面内状态路径，例如：

- 首页 Feed：`window.__INITIAL_STATE__.feed.feeds`
- 搜索结果：`window.__INITIAL_STATE__.search.feeds`
- 详情页：`window.__INITIAL_STATE__.note.noteDetailMap`

还强调使用 `world: "MAIN"` 在页面主 world 执行脚本。

### 第二份

第二份也讲 `__INITIAL_STATE__`，但更偏抽象表述：

- 从详情页 HTML 提取内嵌状态
- 从状态树抽统一字段
- 必要时用浏览器兜底继续读取

没有把“在哪个页面、读哪个状态路径、怎么执行脚本”展开到实现级。

## 2.4 对交互逻辑的描述深度不同

### 第一份

把交互写成了实现能力：

- 页面导航
- 等待加载
- 等待 DOM 稳定
- 查找按钮
- 点击、滚动、输入
- 再次读取状态

尤其评论区，第一份把它写成“观察状态 -> 决定动作 -> 执行动作 -> 再观察”的状态机。

### 第二份

第二份提到评论、详情、列表，但没有深入到这种页面行为控制粒度。

它更像流程架构，而不是页面自动化设计说明。

## 2.5 对批量采集的抽象程度不同

### 第一份

第一份提到：

- 缓存
- 队列
- 冷却
- 统一速率控制器
- 风控退避

但主要是“建议应该这样演进”。

### 第二份

第二份已经把批量同步讲成了一套完整模型：

- 列表页拿候选 ID
- 去重过滤
- 按 ID 逐条抓详情
- 下载媒体
- 落本地知识库
- 更新进度状态
- 下次只抓新增

这部分第二份明显更完整。

## 2.6 对产物落地的关注点不同

### 第一份

重点在采集和浏览器执行，较少展开“最后如何沉淀成内容资产”。

### 第二份

明确包含：

- 图片 / 视频下载
- 本地命名与保存
- Markdown 输出
- vault / 本地知识库沉淀

所以第二份更偏“采集系统 + 归档系统”。

## 2.7 对风控的建模方式不同

### 第一份

更贴近当前代码和页面现象：

- `NoFeedsError`
- `NoFeedDetailError`
- `NotLoggedInError`
- `PageNotAccessibleError`
- `RateLimitError`
- 扫码验证
- 页面不可访问

它更像“页面执行时会遇到哪些具体失败”。

### 第二份

更偏系统级状态建模：

- `ok`
- `retryable_error`
- `risk_limited`
- `auth_invalid`
- `manual_required`

它更适合调度器、限速器和任务状态机。

## 3. 第一份比第二份少了哪些内容

这里的“少了”指覆盖面更弱或系统化程度不足，不是说第一份差。

## 3.1 少了“静态优先，浏览器兜底”的两级抓取策略

第一份基本默认浏览器是主路径。  
第二份明确提出：

- Level 1：直接请求详情页 HTML，解析内嵌状态
- Level 2：浏览器 / CDP 兜底

这让系统更容易做服务端执行，也更节省浏览器资源。  
第一份没有把这套两级策略写清楚。

## 3.2 少了输入归一化层

第二份开头就把输入标准化当成独立步骤：

- 完整链接
- note id
- 标准详情页 URL
- canonical URL

第一份基本没有展开这一层。

## 3.3 少了“列表发现 -> 详情采集 -> 资源下载”这条完整数据流

第二份把采集流程拆得很完整：

`输入 -> 归一化 -> 列表发现 -> 去重过滤 -> 详情抓取 -> 字段抽取 -> 媒体下载 -> Markdown 落盘 -> 更新状态`

第一份更偏单次页面执行和单能力拆解，没有把整个同步链路串得这么完整。

## 3.4 少了本地状态驱动的增量同步模型

第二份明确要求维护：

- `seen_ids`
- `last_cursor`
- `last_success_at`
- 失败原因
- 来源关系

并强调“没有新增就不要继续抓详情”。  
第一份提到了缓存和回退，但没有把增量同步模型落到这个粒度。

## 3.5 少了任务分级和成本分层

第二份明确把任务拆成不同成本层级：

- Level A：正文和基础元数据
- Level B：图片
- Level C：视频 / Live 图
- Level D：评论
- Level E：二级评论 / 转录

并建议默认只抓轻量层。  
第一份没有把“默认抓哪些、哪些按需补抓”写成明确分级策略。

## 3.6 少了媒体下载与知识库落盘链路

第二份对以下内容写得更明确：

- 图片 / 视频下载器
- 本地命名与去重保存
- Markdown 生成
- 本地知识库沉淀

第一份对最终产物形态覆盖较少。

## 3.7 少了调度器 / 限速器 / 状态存储的组件化定义

第二份把这些组件明确拆出来了：

- `Session Provider`
- `List Fetcher`
- `Detail Fetcher`
- `Extractor`
- `Asset Downloader`
- `State Store`
- `Scheduler / Rate Controller`

第一份也谈组件分层，但更偏 bridge 和页面执行，不如第二份完整。

## 4. 第二份比第一份少了哪些内容

第二份更宏观，但少了不少“直接对应当前仓库实现”的落地细节。

## 4.1 少了当前仓库真实 bridge 架构

第一份基本就是围绕当前仓库代码写的，因此开发同学可以直接对照代码理解。

所以：

- 第一份更适合解释“现在这套代码怎么工作”

## 5. 最后结论

可以把这两份文档理解成两个层次：

### 第一份的价值

适合做：

- 当前实现说明
- bridge / 扩展 / 浏览器执行机制说明
- 页面状态提取方法说明
- 页面等待与交互策略说明

### 第二份的价值

适合做：

- ABO 总体采集架构说明
- 批量同步和增量更新设计
- 本地状态、缓存、队列、限速、熔断设计
- 媒体下载与知识库落盘设计

### 最简结论

- 第一份强在“贴近当前代码，讲清楚浏览器内采集怎么实现”
- 第二份强在“把采集当成长期同步系统来设计”

再直白一点：

- 第一份缺第二份的：静态优先策略、输入归一化、增量同步模型、任务分级、资源落盘、调度与状态存储
- 第二份缺第一份的：bridge 架构、扩展执行模型、页面主 world 读取细节、等待/交互机制、评论状态机、页面级异常判断

如果 ABO 后续继续整理文档，最合理的做法不是二选一，而是分层保留：

- 用第一份解释“当前实现怎么跑”
- 用第二份规定“系统后续怎么演进”

## 6. 如果要给第二份实现打上第一份的补丁，怎么缓解当前 IP / 访问频繁限制

如果目标不是推翻第二份，而是“在第二份的总体架构上，吸收第一份里更稳的浏览器内采集能力”，那最合理的补丁方式不是把第二份全部改成浏览器主路径，而是把第一份补成一个更强的 `Level 2`。

也就是：

- 保留第二份的主框架：输入归一化、列表/详情拆分、状态存储、增量同步、限速、熔断
- 用第一份补强第二份的兜底链路：真实浏览器、已登录态复用、页面主 world 读状态、必要交互后再取状态

这样做的原因很直接：

- 当前被 IP / 访问频繁限制，本质上说明“纯后端详情页请求”这条链路对平台来说过密、过直、过重复
- 第一份的优势不是“更能绕过限制”，而是“更像真实用户页面访问上下文”，并且能够直接消费页面已经拿到的数据
- 第二份的优势仍然要保留，因为真正降风险的核心还是：少抓、去重、缓存、限速、熔断

所以正确补丁不是“只换抓取手段”，而是：

`第二份的系统调度层 + 第一份的浏览器执行层`

### 6.1 建议补丁后的抓取优先级

推荐把第二份原来的两级策略改成下面这个顺序：

#### Level 1：缓存命中直接返回

先查：

- 详情缓存
- 列表缓存
- 最近访问记录
- 失败冷却记录

如果目标在短窗口内已经抓过，就不要再次打开详情页。

#### Level 2：静态详情页抓取

仅在预算允许时，尝试：

- 请求详情页 HTML
- 解析 `__INITIAL_STATE__`
- 抽取轻量元数据

如果成功，直接返回，不再进入浏览器。

#### Level 3：真实浏览器兜底

当出现以下情况时，再切到第一份的方案：

- 静态详情页拿不到完整状态
- 详情页命中访问限制文案
- 后端链路连续失败
- 当前内容必须依赖真实登录上下文

此时：

- 复用同一个真实浏览器会话
- 通过 bridge / CDP 进入已有 tab 或受控 tab
- 在页面主 world 读取状态树
- 只有确实需要时才做点击、滚动、输入

这个顺序的好处是：

- 正常场景仍然保留第二份的轻量路径
- 风险升高时，自动切到第一份更贴近真实用户环境的读取方式
- 不会把所有请求都压到同一种访问模式上

### 6.2 应该从第一份补到第二份的具体能力

建议补下面五类能力。

#### 补丁 1：真实浏览器会话池

把第二份里的 `Session Provider` 从“只管理 cookie / CDP 连接”升级成“真实浏览器上下文提供者”。

至少要支持：

- 固定复用已登录浏览器 profile
- 一个批次尽量复用同一会话
- 记录会话最近风险状态
- 会话进入冷却后禁止继续派发任务

这能减少“每条内容都像新客户端”带来的噪音。

#### 补丁 2：页面主 world 状态提取器

把第一份的核心经验补进第二份的 `Detail Fetcher` / `Extractor`：

- 优先取页面全局状态对象
- 读取 `window.__INITIAL_STATE__` 及同类首屏状态
- 在页面主 world 执行脚本，而不是只依赖服务端 HTML 解析

这样做的收益是：

- 少造一轮额外请求
- 少追内部接口
- 能直接利用页面已经完成的登录态和前端渲染结果

#### 补丁 3：等待与交互策略组件

第二份现在更像系统设计，需要补上第一份这种操作级能力：

- 等待页面加载完成
- 等待状态对象出现
- 等待 DOM 稳定
- 点击后等待结果出现
- 只有必要时才滚动、展开、输入

重点不是“多操作”，而是“把动作降到最少，只做拿数据真正必要的动作”。

#### 补丁 4：页面级风控探针

把第一份里的页面失败识别补到第二份的风险控制器前面，做成页面级探针。

至少识别：

- 安全访问限制
- 访问频繁
- 扫码验证
- 登录失效
- 页面不可访问
- 内容不存在 / 私密

然后再映射到第二份的系统状态：

- `retryable_error`
- `risk_limited`
- `auth_invalid`
- `manual_required`

这样第二份的调度器才不会只看到“抓取失败”，而能知道“为什么失败”。

#### 补丁 5：评论和重任务的状态机化执行

对于评论、楼中楼、展开更多这类高成本能力，不要继续按“请求一个重任务就直接全开”的方式做。

应借用第一份的模式：

`观察状态 -> 决定动作 -> 执行动作 -> 再观察`

并且明确要求：

- 默认不抓评论
- 必须抓时单独派发任务
- 每次只推进一个小步骤
- 连续停滞就退出，不做无限滚动

这会明显降低重任务造成的访问膨胀。

### 6.3 对“当前 IP 被限制”最有帮助的改造顺序

如果现在已经在被限制，优先级建议按下面顺序落地，而不是一次性大改。

#### 第一优先级：把浏览器内兜底正式接进第二份

先做：

- 静态详情页失败后自动切浏览器会话
- 浏览器侧读页面状态树
- 同一批任务复用同一浏览器上下文

这一步的目的不是提速，而是降低“纯后端连续详情页请求”的暴露度。

#### 第二优先级：把重任务默认关闭

默认仅抓：

- 标题
- 正文
- 作者
- 发布时间
- 基础互动数

默认关闭：

- 评论
- 二级评论
- 视频补抓
- Live 图补抓
- 转录

先把详情主链路压到最轻。

#### 第三优先级：把失败后的处理从“重试”改成“切路径或冷却”

命中以下信号时：

- 访问频繁
- 安全限制
- 页面异常

不要继续后端重试，应改成：

1. 记录风险事件
2. 立即停止当前静态链路
3. 视预算切浏览器兜底，或直接进入冷却
4. 冷却期内不再重复请求同目标

#### 第四优先级：把详情抓取改成“候选收集 + 低速消费”

也就是：

- 列表阶段只拿 ID
- 详情阶段按预算慢速消费
- 消费过程中优先抓新内容
- 旧内容只做低频补抓

这样可以避免某个批次瞬时打开大量详情页。

### 6.4 补丁后的最终实现建议

如果把这次补丁落成一句话，建议写成：

> 第二份继续保留“静态优先、增量同步、状态驱动、风控熔断”的总体架构；同时引入第一份的“真实浏览器会话复用、页面主 world 状态读取、页面级等待与交互控制”作为正式兜底层，用来替代当前高风险的纯后端重复详情页访问。

这样改的结果不是“绕过 IP 限制”，而是把系统从单一、重复、容易触发限制的访问模式，改成：

- 更少的重复访问
- 更明确的轻重任务分级
- 更真实的页面上下文兜底
- 更早的风控识别与熔断

从工程角度看，这才是第二份吸收第一份后，对“当前被 IP 限制”最有实际帮助的补丁方向。
