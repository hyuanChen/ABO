# 小红书专辑抓取 Memory

## 这份文档的用途

这份文档不是正式设计文档，而是当前阶段的小红书收藏专辑抓取“工作记忆”。

目标是方便后续快速 recall：

- 现在专辑抓取到底是怎么做的
- 中间踩过哪些坑
- 哪些问题已经确认
- 哪些问题还没彻底解决

## 当前代码状态

当前代码已经回滚到比较接近“原来效果”的专辑抓取逻辑：

```text
后台无界面 Playwright
-> 打开专辑页
-> 读取 window.__INITIAL_STATE__.board.boardFeedsMap[board_id].notes
-> 继续滚动页面
-> 再次读取同一个 state
-> 合并去重
-> 连续几轮没有明显增长后停止
```

也就是说，当前主路径仍然是：

- 后台无界面
- 不弹出浏览器
- 不要求用户手动打开可见页面
- 页面自己滚动加载
- 我们读取页面 state

不是：

- 纯后端直连翻页接口
- 主动重放签名请求
- 要求打开可见浏览器窗口

## 当前前端 UI 状态

### 收藏整理页

保留的核心能力：

- 后台获取收藏专辑
- 多选专辑
- 全量 / 增量抓取
- 最近 N 天
- 下载图片 / 下载 Live 图 / 下载视频
- 记录评论（测试中，需要打开浏览器页面）
- 随机间隔 8-15 秒

### 进度区

当前进度区在 `收藏专辑抓取` 卡片内部，位置已经放到抓取选项正下方。

进度显示内容包括：

- 当前专辑
- 专辑进度
- 已加载条数 / 预期条数
- 已翻页次数
- 新增 / 跳过 / 失败
- 当前专辑笔记进度
- 等待限速秒数

没有再显示后来新增的：

- 接口响应次数
- 连续空转轮数

这些“额外调试信息”已经回滚掉了。

## 当前默认行为

### 抓取模式

- 默认是 `全量`
- `全量` 会忽略最近天数限制
- `增量` 才会根据本地 `.xhs-albums-progress.json` 跳过已抓过的笔记

### 最近天数

- UI 默认是 `180`
- 输入可以删空
- 真正提交抓取时，如果是增量并且为空，会按 `180` 天处理
- 如果是全量，则忽略最近天数

### 限速

- 每条详情页抓取后随机等待 `8-15` 秒
- 默认上限值现在是 `12`
- 如果命中 `300012 / 300013 / 安全限制 / 访问频繁`，任务会直接停下

### 后台会话复用

- 一次多专辑抓取任务内，会复用同一个后台无界面 browser/context/page
- 不会为每个专辑重新起一个新的浏览器会话
- 这样可以减少登录态抖动，也更接近页面自然使用行为

## 当前保存路径

### 专辑抓取

专辑抓取现在保存到专辑子文件夹：

```text
<vault>/xhs/<专辑名>/
```

里面会有：

- Markdown
- `img/`
- `video/`

### 非专辑抓取

普通单条抓取、批量链接抓取，仍然保存在：

```text
<vault>/xhs/
```

不会进入专辑子目录。

## 当前缓存与进度文件

### 专辑缓存

```text
<vault>/xhs/.xhs-albums-cache.json
```

用途：

- 保存最近一次读取到的专辑列表
- 页面刷新后专辑不消失
- 后台专辑列表读取失败时，可以优先恢复本地缓存

### 专辑进度

```text
<vault>/xhs/.xhs-albums-progress.json
```

用途：

- 记录每个专辑的 `seen_note_ids`
- 记录每条笔记属于哪些专辑
- 供增量抓取使用

## 已确认的事实

### 1. 后台 headless 能拿到专辑列表

已经验证过：

- 后台无界面 Playwright 能成功进入收藏 -> 专辑
- 可以拿到专辑名、数量、封面图、board_id

### 2. 专辑页笔记列表不是总能全量吐出来

已经实测过：

- 有一个 `故事` 专辑，页面显示总数是 `193`
- 之前有一版抓取逻辑，最终只能拿到 `141`

这说明：

- 问题不是纯前端显示错
- 而是专辑页自然加载出来的数据本身就可能不完整

### 3. 真实翻页接口存在

已经定位到专辑翻页接口：

```text
https://edith.xiaohongshu.com/api/sns/web/v1/board/note
```

并确认：

- 页面自然滚动时会触发它
- 它确实返回 cursor + notes
- 页面里存在签名函数 `window._webmsxyw`

### 4. 直接主动重放翻页接口还不稳定

已验证到：

- 纯后端直连请求会 `406`
- 页面内主动 `fetch` 也可能 `406 / 461 / 300011`
- 即使补了部分签名头，也未必能复现页面自然请求

所以目前还没有把主逻辑切到“主动 cursor 翻页”。

## 已经回滚掉的额外逻辑

后面尝试过一些更激进的补抓方案，但目前已经回滚：

- “未达到专辑总数前绝不停止”的强约束
- 用真实接口响应计数作为主停止条件
- 首屏空时强制 reload 的额外补丁
- 进度 UI 中显示接口响应次数 / 连续空转轮数

回滚原因：

- 你希望回到“原来效果”的思路
- 不要继续引入额外复杂逻辑
- 保持后台无界面 + 页面自己滚动加载为主

### 这次具体删掉的功能列表

为了回到“原来效果”的专辑抓取逻辑，这次明确删掉的是：

1. `未达到专辑总数前不会主动停止`
2. `board/note 真实接口响应监听并合并 notes`
3. 前端进度中的：
   - `接口响应：x 次`
   - `连续空转 x 轮`
4. `首屏空时强制 warmup / reload` 的补丁逻辑
5. “接口响应 + state”联合决定停止的那套更复杂的停止条件

删掉后的当前逻辑重新回到：

```text
后台无界面页面
-> 读取 state
-> 页面自己滚动
-> 再读 state
-> 连续几轮没明显增长就停
```

## 当前还存在的不确定点

### 1. 为什么有些专辑还是抓不全

当前最核心的不确定点是：

- 页面自己滚动时，为什么有些专辑明明显示 193，但 state 只增长到中间值

可能原因包括：

- 小红书在当前风控环境下，没有继续吐后续页
- 页面继续下拉后没有再触发新的数据写入
- 某些旧帖 / 已删帖 / 权限异常帖子没有真正下发详情

### 2. 专辑总数和实际可抓数可能对不上

现在 UI 已经提示：

```text
因帖子删除，总数可能对不上
```

这不是装饰文案，是真实可能发生的情况。
