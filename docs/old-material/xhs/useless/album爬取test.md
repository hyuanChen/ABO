# 小红书专辑爬取测试记录

## 背景

这份文档只记录“小红书收藏专辑抓取”里，专辑内笔记列表是怎么加载出来的。

重点回答两个问题：

1. 一开始的专辑滚动加载思路是什么。
2. 后来为了补全数量，又尝试了什么新思路。

这里说的“打开页面”，都指 **后台无界面 headless 页面**，不是弹出给用户看的浏览器窗口。

## 第一版：原来的加载思路

最初的思路比较直接：

```text
进入专辑页
-> 等页面首屏渲染
-> 读取 window.__INITIAL_STATE__.board.boardFeedsMap[board_id].notes
-> 继续滚动页面
-> 再次读取同一个 state
-> 把新出现的 notes 合并去重
-> 连续几轮没有新增后停止
```

对应的核心点：

- 主数据源是 `window.__INITIAL_STATE__`
- 页面滚动只是为了触发前端继续往 state 里补数据
- 不主动请求接口
- 不依赖外部浏览器
- 后台 headless 就能工作

### 这一版的优点

- 逻辑最简单。
- 最接近“原来效果”。
- 不需要主动模拟小红书接口签名。
- 如果专辑页本身会老老实实往 state 里补数据，这一版最稳。

### 这一版的问题

- 有些专辑页只把首屏或前几屏数据写进 state。
- 页面明明还有更多收藏，但继续滚动时，state 不一定继续增长。
- 所以会出现：
  - 专辑总数是 193
  - 实际只读到 33 / 108 / 141 这类中间值

## 第二版：后来的优化思路

后来为了补数量，做了两层增强。

### 2.1 首屏预热增强

先不是一上来就猛滚，而是：

```text
进入专辑页
-> 等 4 秒
-> 循环检查首屏 state 是否已经出来
-> 如果还没出来，轻微滚动一点再等
-> 如果还是空，后台 reload 一次
-> 只有首屏数据出现后，才开始后续翻页
```

这个修改主要是解决：

- headless 环境里，专辑页 hydration 比可见浏览器慢
- 太早开始滚动会把“首屏还没出数据”的页面误判成空
- 于是会出现 `已加载 0/76`、`接口响应 0` 这种假空状态

### 2.2 滚动 + 真实响应监听

后来的第二层增强不是只看 state，而是：

```text
进入专辑页
-> 后台监听 /api/sns/web/v1/board/note 响应
-> 一边滚动页面
-> 一边继续读取 state
-> 把“state 里的 notes”和“接口响应里的 notes”一起合并去重
-> 连续很多轮既没有新 state，也没有新接口响应，才停止
```

对应代码思路：

```text
_fetch_board_notes_headless
-> page.on("response") 监听 board/note
-> merge_notes(response notes)
-> merge_notes(state notes)
-> idle_rounds 很多轮后才停
```

### 这一版的目的

- 尽量保留原来“靠页面自己加载”的方式
- 但不再只盯 `window.__INITIAL_STATE__`
- 如果专辑页继续滚动时实际发了翻页接口，就把接口里的 notes 也接住

### 这一版观察到的现象

实际测试 `故事` 专辑时，拿到了下面这种情况：

```text
专辑总数：193
首屏 + 翻页接口合并后：141
真实翻页接口自动返回次数：4 次
```

这说明：

- 之前看到的 `141 / 193` 不是前端进度条算错
- 而是专辑页自己只继续吐到了 141 左右
- 再继续滚动，页面也没有再发新的翻页响应

## 为什么没有直接切成“主动请求接口翻页”

已经验证到：

- 专辑页继续加载时，真实接口是  
  `https://edith.xiaohongshu.com/api/sns/web/v1/board/note`
- 这个接口确实存在 cursor 翻页
- 页面里也能拿到签名函数 `window._webmsxyw`

但是直接在后台主动请求时，会遇到：

- 纯后端请求：`406`
- 页面内手工 `fetch`：`406` / `461`
- 即使补了 `x-s / x-t / x-s-common`，也可能被 `300011` 拦住

也就是说：

- 页面自然触发的翻页请求能成功
- 我们手工重放同样的请求，不一定能成功

所以当前还没有把主路径切成“主动请求接口翻页”，而是优先保留：

```text
后台无界面页面
-> 让页面自己滚动
-> 让页面自己触发合法请求
-> 我们只监听它返回了什么
```

## 当前结论

### 原来的思路

- 靠页面 state
- 滚动触发 state 增长
- 最简单，也最符合最初效果

### 后来的思路

- 仍然是后台无界面
- 仍然是页面自己滚动加载
- 但同时监听真实翻页响应
- 再把 state 和响应结果合并

### 为什么还会抓不全

如果专辑页自己只继续发到某个 cursor 就停了，那么：

- 再怎么滚
- 再怎么等
- 我们都只能拿到页面真正吐出来的那一部分

这种情况下，问题不再是“滚动策略太弱”，而是：

- 小红书这个专辑页的自然加载链路，在当前环境下没有继续吐后续数据

## 后续方向

如果后面还要继续补全，可以分两条线：

### 方案 A：坚持原效果优先

- 继续用后台无界面页面
- 继续让页面自然滚动
- 调整等待节奏、滚动幅度、触底回拉方式
- 不碰主动签名请求

适合目标：

- 尽量保留和原来一致的行为
- 不增加主动请求的风险

### 方案 B：做专门的 cursor 补抓链路

- 先让页面自然跑出前几页
- 记录成功请求里的 cursor、headers、签名参数
- 再尝试在同一个页面上下文里补后续 cursor

适合目标：

- 专门冲“193 就要尽量接近 193”
- 接受实现复杂度和维护成本更高

## 当前建议

如果目标是“先恢复之前可用的体验”，建议优先采用：

```text
原来的页面滚动加载思路
+ 首屏预热
+ 真实翻页响应监听
```

也就是：

- 不打开可见浏览器
- 不要求用户手动点页面
- 后台 headless 继续做
- 以页面自然加载为主
- 监听真实接口作为辅助

这就是现在代码正在收敛的方向。

## 本次已删除 / 已回滚的功能

这次为了回到“原来效果”的专辑抓取方案，已经明确删掉或回滚了下面这些额外逻辑：

1. `未达到专辑总数前不会主动停止`
   - 之前尝试过把专辑卡片上的总数当成硬目标。
   - 现在已取消，不再要求一定追到 `count` 才停。

2. `真实翻页接口响应计数驱动`
   - 之前在 headless 页面里监听 `/api/sns/web/v1/board/note`。
   - 还把接口返回的 notes 和页面 state 里的 notes 一起合并。
   - 现在这条额外链路已经回滚。

3. `response_hits / idle_rounds 调试状态`
   - 前端进度里一度展示过：
     - 接口响应次数
     - 连续空转轮数
   - 现在这些调试信息已经删除。

4. `首屏空时的额外 warmup / reload 补丁`
   - 后面一版里，为了避免 `0/76`，增加了：
     - 首屏多轮预热
     - 首屏为空时 reload 一次
   - 现在也已经回滚，不保留这层额外补丁。

5. `按接口响应驱动的滚动停止策略`
   - 之前停止条件不再只看 state 是否增长，
     而是看“接口响应 + state”是否同时停住。
   - 现在已回滚成更早那版：
     - 主要看 `window.__INITIAL_STATE__`
     - 页面自己滚动
     - state 连续几轮没有明显增长后停止

## 当前保留的功能

当前仍然保留的核心行为是：

- 后台无界面 Playwright
- 不弹出可见浏览器
- 多选专辑后，按专辑顺序一个个爬
- 每个专辑内部再逐条抓笔记
- 专辑抓取结果保存到对应专辑文件夹
