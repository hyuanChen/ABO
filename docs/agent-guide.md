# ABO Agent Guide

这份文档写给第一次接手 ABO 的开发者、代理和协作者。  
目标不是解释所有细节，而是帮助你在最短时间内进入正确的开发心智：知道这个项目现在是什么、应该先看什么、从哪里下手，以及怎样继续做出和现有产品一致的改动。

如果你是来继续“vibe coding”的，这份文档就是你的入口。

---

## 1. 先用一句话理解 ABO

ABO，`Another Brain Odyssey`，不是一个普通的桌面应用，也不是单纯的爬虫、笔记工具或聊天工具。

它是一个本地优先的个人智能系统，用来把这些东西重新收回到自己手里：

- 论文
- 平台内容
- 收藏与关注
- 今日情报
- 文献库与情报库
- Wiki
- 手记
- 个人状态与成长记录

它真正做的事，是把用户每天输入的内容，逐步整理成一个可以反复调用、不会轻易遗忘、还能继续参与判断和成长的“另一个大脑”。

---

## 2. 先理解三条主链，再写代码

ABO 当前最重要的产品结构，不是某一个页面，而是三条主链：

### 2.1 研究链

`论文发现 -> 今日情报 -> 阅读筛选 -> 文献库 / Wiki -> 助手生成判断与 idea`

这条链处理的是研究、论文、Follow Up、阅读、知识沉淀和研究输出。

### 2.2 注意力链

`平台输入 -> 热点归类 -> 偏好记录 -> 情报库 -> 日报 / 长期注意力画像`

这条链处理的是小红书、B 站、平台热点、兴趣偏好、生活输入和注意力轨迹。

### 2.3 成长链

`个人记录 -> 状态总结 -> 规律识别 -> 建议输出 -> 虚拟陪伴 / 自我建模`

这条链处理的是角色主页、手记、健康、状态、建议、长期画像和陪伴感。

### 2.4 你在开发前必须回答的两个问题

无论用户提什么需求，先回答：

1. 这个需求属于哪条主链？
2. 它最后沉淀到哪里？

常见沉淀位置包括：

- `今日情报`
- `情报库`
- `文献库`
- `Wiki`
- `角色主页`
- `手记`
- `数据洞察`

如果一个需求答不清这两点，说明它很可能还只是一个孤立想法，不适合直接开工。

## 3. 当前仓库的真实状态

开始之前，不要把仓库想得过于整洁和统一。当前真实情况是：

### 3.1 代码和文档并不完全同步

仓库里有很多高质量文档，但不是所有文档都和今天的代码完全一致。  
遇到冲突时，以当前代码为准，文档用于理解思路和约束。

### 3.2 文档目录正在重组

你会看到：

- `docs/` 下只保留少量当前还在用的文档

- `docs/chat-guidance/` 和 `docs/skills/` 里放的是更偏专项的积累

这意味着：

- 但它们默认不是新功能的最终落点

### 3.3 仓库里有多阶段遗留

当前项目里同时存在：

- 旧的 Claude 命名与桥接痕迹
- 现在的 Codex / assistant 工作面
- 旧的 canvas / xhs 设计稿
- 新的文档结构和计划文件

这很正常。不要一上来就想大统一重构。

### 3.4 有些页面已经可用，有些仍在演化

相对更成熟的方向：

- 小红书主动工具链
- Bilibili 主动工具链
- 论文追踪与文献库路径
- 今日情报 / 模块管理 / Wiki 方向
- 助手工作区的上下文组织方式

需要谨慎判断成熟度的方向：

- 健康管理
- 手记的数据接入完整度
- 部分仪表板数据项
- 旧命名、旧路由、旧文档中残留的能力描述

所以写代码前，先看页面是不是已经真的接上真实链路。

## 5. 先从哪里上手开发

### 5.1 第一步：把项目跑起来

常用命令：

```bash
npm install
python3 -m pip install -r requirements.txt
npm run tauri:fresh-dev
```

如果你只想分开调试：

```bash
python3 -m abo.main
npm run dev
```

第一次启动后，建议真的走一遍 UI：

1. 配置情报库 / 文献库路径
2. 打开小红书工具或 Bilibili 工具
3. 跑一次手动工具链
4. 去今日情报看卡片
5. 去情报库 / 文献库看保存结果
6. 再去 Wiki / 助手看后续使用路径

如果你没有亲自走过这条链，很容易在错误层级上改需求。

### 5.2 第二步：确认需求落在哪个现有入口

ABO 的一级心智已经在侧边栏里写出来了：

- `角色主页`
- `助手`
- `今日情报`
- `数据洞察`
- `情报库`
- `文献库`
- `Wiki`
- `手记`
- `模块管理`
- `小红书工具`
- `哔哩哔哩工具`
- `论文追踪`
- `健康管理`

优先把新功能接到已有入口里，而不是新开一个孤立页面。

### 5.3 第三步：优先扩现有链，不要新造平行链

ABO 最重要的开发习惯之一是：

> 先找已有主动工具、已有保存链、已有模块配置、已有展示卡片，再决定要不要新增。

特别是对这些需求：

- 定时任务
- 模块监控
- 今日情报卡片
- 保存到库
- 保存到 Wiki
- 助手读取上下文

如果主动工具已经有完整链路，定时和 Feed 必须复用它，而不是复制一份简化版。

### 5.4 第四步：把 Wiki 当作“LLM 持续维护的编译层”

你现在的 Wiki 维护方式，不应该按传统文档系统去理解，而应该按 `LLM Wiki` 模式理解。

当前可直接参考：

- `docs/LLM-wiki.md`

它背后的核心心智是：

`raw sources`
-> `LLM 持续提取、归并、交叉引用`
-> `形成持久 Markdown wiki`
-> `用户通过 Obsidian / UI / 助手继续查询与扩写`

这意味着：

1. Wiki 不是单纯展示 Markdown 的页面
2. Wiki 也不是 query-time RAG 的附属结果
3. 好的分析、总结、对比和关系发现，应该考虑是否写回 Wiki
4. LLM 在这里的角色不是一次性回答问题，而是持续维护一个可增长的知识层

如果你在做任何与 Wiki 相关的开发，先想清楚它属于哪一段：

- `ingest`：把新材料变成 wiki 更新
- `query`：围绕现有 wiki 回答问题
- `lint`：检查冲突、孤页、缺链和过时内容

## 6. 用什么心智继续“vibe coding”

ABO 适合的开发方式不是大而全设计，而是 `薄切片端到端推进`。

### 6.1 什么叫薄切片

一个好的小切片，至少要包含：

1. 用户从哪进入
2. 用户看到了什么反馈
3. 数据最后存到哪里
4. 后续能不能被 Feed / Wiki / 助手继续用

如果你只做了一个按钮、一个对话框或一个接口，但没接到后续链路，通常不算完成。

### 6.2 ABO 更适合怎么推进

推荐顺序：

1. 先把最短的手动路径做通
2. 再把通用逻辑抽出来
3. 再接到模块 / 调度 / Feed
4. 再接到 Wiki / 助手 / 洞察

也就是：

`先能用`
-> `再复用`
-> `再自动化`
-> `再接入长期系统`

### 6.3 不推荐的推进方式

不建议一上来就：

- 全面重构目录
- 强行统一所有命名
- 先做宏大架构图再动最小链路
- 为定时任务新写一套抓取逻辑
- 先补很多抽象，再去接 UI

ABO 当前阶段更适合“贴着真实使用路径补强”，而不是“为未来想象的大系统做大规模预重构”。

## 7. 这个仓库里最重要的工程原则

### 7.1 本地沉淀优先

ABO 的核心价值不在 UI，而在这些长期资产：

- `情报库`
- `文献库`
- `Wiki`
- `~/.abo/*`
- `cards.db`

当你在改功能时，要始终问：

> 这次改动让什么长期资产变得更稳定、更清晰、更可复用？

### 7.2 主动工具是事实来源

对采集类能力来说，手动工具页是事实来源。  
Feed、调度、模块管理、摘要、洞察都应该建立在主动工具已经跑通的链路上。

### 7.3 侧边栏不是装饰，是 IA

侧边栏已经表达了当前产品的信息架构。  
新增能力时，优先顺着这个 IA 增量扩展，而不是临时拼入口。

### 7.4 库和 Wiki 是沉淀层，不是附属品

很多产品会把“保存”当成一个结尾动作。  
ABO 不是。对它来说，保存只是下一轮工作的开始。

真正好的功能，不是把结果展示出来，而是让结果后续还能继续：

- 被回看
- 被筛选
- 被连接
- 被讨论
- 被总结

## 8. 常见开发路径

### 8.1 你要接一个新来源

推荐路径：

1. 先做主动工具页
2. 先保证能预览
3. 再保证能保存到正确目录
4. 再把配置纳入统一模块配置
5. 再接模块管理
6. 再接定时任务
7. 再让它进入今日情报和助手上下文

### 8.2 你要做一个新模块

推荐路径：

1. 先看现有模块实现和 `abo/sdk/*`
2. 用现有 `Module` 结构
3. 先定义抓什么、如何处理、保存到哪里
4. 先做可手动验证的输出
5. 再注册到模块管理和调度

如果你是第一次写模块，优先直接读这些真实代码：

- `abo/sdk/base.py`
- `abo/sdk/types.py`
- `abo/runtime/runner.py`
- `abo/modules/routes.py`
- 任意一个当前仍在使用的模块实现

### 8.3 你要做一个新助手能力

推荐路径：

1. 先确认上下文来自哪里
2. 明确它服务于研究链、注意力链还是成长链
3. 优先做“模板化工作流”，不要只做空白聊天
4. 输出要能回写到库、Wiki、手记或待办

### 8.4 你要做一个新页面

推荐路径：

1. 先确认它是不是应该成为新一级入口
2. 如果不是，优先挂到现有页面中
3. 使用现有 `Layout`、`store`、`api` 模式
4. 保持和现有页面一致的反馈节奏与卡片语言

## 9. 现在最值得告诉下一个开发者的“坑”

1. `不要把旧画布当成当前代码`
   旧 canvas 很有产品启发，但不能直接代表当前实现。

2. `不要恢复用户正在迁移的文档`
   当前 docs 明显在重组，看到大量删除不要自行还原。

3. `不要只做前端表面`
   ABO 很多功能看起来像 UI 功能，实质上是保存链、沉淀链、后续使用链。

4. `不要忽略目录和路径`
   情报库、文献库、Wiki 的保存路径错了，功能基本就坏了。

5. `不要平行实现抓取链`
   这是最严重的坑，后期会让工具页、定时、Feed 三套逻辑分裂。

6. `不要假设所有页面都已接入真实数据`
   有些页面现在是方向明确、数据层还在继续补。

7. `不要把 ABO 做成一个“全自动黑箱”`
   这个产品的价值很大一部分来自用户自己的判断、筛选和沉淀动作。

## 10. 你改完之后，至少要验证什么

### 采集 / 监控类需求

- 主动工具页能跑
- 结果预览正确
- 保存位置正确
- 今日情报里能看到
- 模块管理状态正确

### 文献库 / 情报库 / Wiki 类需求

- 本地文件真的被写出来
- 页面能重新读回来
- 后续页面能继续消费这些结果
- 如果结果本质上是一段长期有价值的分析，要判断它是否应该写回 Wiki，而不是只停留在聊天记录里

### 助手 / 角色主页 / 洞察类需求

- 上下文来源真实
- 输出不是孤立文字，而能继续被系统利用
- 至少和一条主链明确连接

如果你需要理解这个项目为什么会长成现在这样，优先看这些材料：

- `README.md`：当前产品总叙事
- `docs/abo-user-guide.md`：当前实际使用路径
- `docs/LLM-wiki.md`：当前 Wiki 维护模式与长期知识层心智

这些材料对“为什么这样设计”“用户更在意什么”“语言风格该怎么贴近产品”很有帮助。

---

## 12. 一句话总结给下一个开发者

在 ABO 里继续开发，最重要的不是“多加一个功能”，而是：

`顺着已有主链，把输入接进来，把沉淀做稳定，把后续调用变顺。`

做到这三点，你写出来的东西通常就会和这个产品长在一起，而不是贴在外面。
