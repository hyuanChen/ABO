# XHS Crawler Overall

本文不是规避平台风控的说明书，而是给内部开发用的合规抓取设计文档。目标是把当前仓库里已经存在的小红书采集实现，抽象成一套可以复用到其他 Agent 和其他内容源的工程逻辑。

如果当前遇到的问题是“访问过于频繁，出现安全访问报警”，本文给出的方向是：

- 降低请求密度
- 减少重复访问
- 用增量同步替代全量重扫
- 在命中风控信号时立即停止任务
- 通过缓存、队列、冷却和人工介入降低风险

不包含也不建议以下做法：

- 绕过平台安全机制
- 伪造、破解或批量规避登录/签名/验证
- 用代理池、指纹伪装、批量账号轮换去对抗平台限制

## 1. 当前实现的主要逻辑

结合仓库现状，现有实现可以概括为 6 个步骤：

1. 输入标准化  
   输入可以是完整链接，也可以是 note id。系统先统一转换成标准详情页 URL。

2. 优先走后端详情页抓取  
   直接请求详情页 HTML，解析页面里的 `window.__INITIAL_STATE__`，从里面提取笔记主数据。

3. 后端抓不到时，用本机已登录浏览器做兜底  
   通过本地 CDP 调试端口连接真实浏览器上下文，读取页面里的前端状态，而不是在后端硬拼更激进的请求链路。

4. 抽取结构化字段  
   从状态树里提取：
   - 笔记基础信息
   - 作者信息
   - 发布时间
   - 标签
   - 图片 / Live 图 / 视频 URL
   - 点赞、收藏、评论、分享等互动计数

5. 下载媒体并写入本地知识库  
   把图片、视频等资源按规则保存到 vault 目录，同时生成 Markdown。

6. 批量模式下做增量同步  
   对收藏夹、专辑、搜索结果等列表来源，先拿到 note id 列表，再逐条抓详情，并把抓取进度落盘，下次只补新增项。

一句话总结：当前实现的核心不是“多快地打接口”，而是“尽量少请求、尽量复用已有登录态、尽量把一次抓取变成长期可增量维护的本地内容库”。

## 2. 现有代码里的关键设计

### 2.1 单帖抓取链路

单帖链路的关键顺序如下：

1. 规范化 URL / note id
2. 发起详情页请求
3. 从 HTML 中提取 `__INITIAL_STATE__`
4. 从 `noteDetailMap` 中定位目标笔记
5. 解析正文、作者、互动、图片、视频等字段
6. 下载需要的媒体
7. 输出 Markdown

这个设计的价值在于：

- 对单条内容来说，请求链路短
- 依赖页面已有状态，避免额外拼很多接口
- 数据结构稳定时，抽取逻辑很容易复用

### 2.2 CDP 兜底

当前实现没有把 CDP 当作默认主路径，而是兜底路径。这个判断是合理的。

CDP 兜底的作用：

- 复用真实浏览器登录态
- 读取页面运行后的前端状态
- 在后端直接请求拿不到完整内容时补救

它解决的是“上下文一致性”问题，不是“高频访问能力”问题。  
换句话说，CDP 可以提高成功率，但不能把它理解成风控绕过手段。

### 2.3 批量抓取不是直接暴力并发

从当前实现看，批量模式的设计重点有三个：

- 先拉列表，再按 note id 逐条处理
- 每批之间做冷却
- 每条之间做随机等待

同时，代码里会把已经处理过的 note id 记录到进度文件，避免重复抓同一批内容。

这说明当前实现的工程思想已经比较明确：

- 不追求极限吞吐
- 优先控制稳定性
- 用状态文件减少重复网络访问

### 2.4 命中风控时立即中断

当前代码已经识别了一些典型风险信号，例如：

- 安全限制
- 访问频繁
- IP 风险
- 登录状态异常

一旦命中这些错误，流程会直接停止，而不是继续重试打满。这一点很关键，应该保留并继续强化。

原因很简单：

- 继续重试通常只会扩大风险
- 短时间内重复失败，说明系统预算已经超限
- 这时应该进入冷却与人工确认，而不是自动冲击平台

## 3. 最值得提炼的通用逻辑

下面这些内容，是当前 XHS 方案里最精华、也最适合抽象给其他 Agent 复用的部分。

### 3.1 两级获取策略：静态优先，浏览器兜底

推荐抽象：

- Level 1：直接请求内容页，解析页面内嵌状态
- Level 2：必要时复用真实浏览器上下文读取状态

适用范围：

- 小红书
- 知乎
- B 站
- 很多前端首屏带预加载状态的站点

这个模式通用、稳、可维护。  
不要一上来就堆很多隐藏接口，也不要默认用复杂自动化浏览器跑全流程。

### 3.2 列表页与详情页分离

通用抓取应该拆成两个阶段：

- 列表发现：拿到内容主键，例如 note id / post id / article id
- 详情采集：基于主键抓具体内容

这样做的好处：

- 列表层可以单独缓存
- 详情层可以单独重试
- 增量同步更简单
- 更容易做幂等和去重

### 3.3 抓取系统必须有本地状态

真正可用的采集系统一定不是“脚本跑一次就算了”，而是“有状态的同步器”。

至少要保存：

- 已抓取主键集合
- 每个主键最后一次成功时间
- 来源关系，例如某条内容来自哪个专辑/收藏夹
- 最近一次任务运行时间
- 最近一次失败原因

这比任何“提速技巧”都更重要。  
因为大多数风险告警，根因不是单次请求不会做，而是重复请求太多、无状态重扫太多。

### 3.4 媒体下载与元数据采集解耦

建议统一拆成两层：

- 元数据层：标题、正文、作者、计数、标签、发布时间
- 资源层：图片、视频、Live 图、转录等大文件

这样可以支持不同任务等级：

- 快速索引模式：只拉元数据
- 完整归档模式：再补媒体

这能显著降低默认请求成本。

### 3.5 风控不是异常分支，而是一级状态

不要把“访问频繁”当成普通报错字符串处理。  
应该把它抽象成明确状态：

- `ok`
- `retryable_error`
- `risk_limited`
- `auth_invalid`
- `manual_required`

一旦状态进入 `risk_limited`，系统应切换策略，而不是继续沿用原来的重试逻辑。

## 4. 对其他 Agent 通用的抓取架构

下面是一套适合给其他 Agent 复用的通用架构。

## 4.1 组件划分

### A. Input Normalizer

职责：

- 统一处理 URL、短链接、主键
- 解析来源类型
- 生成标准资源标识

输出示例：

```text
source=xhs
entity_type=note
entity_id=<note_id>
canonical_url=<normalized_url>
```

### B. Session Provider

职责：

- 提供合法登录态
- 管理 cookie / 浏览器上下文
- 对接本机 CDP 或受控浏览器

注意：

- 会话是“稳定性资源”，不是“绕过资源”
- 一个任务应尽量复用同一上下文，减少新建会话

### C. List Fetcher

职责：

- 拉取列表页或集合页
- 抽取内容主键
- 返回增量候选集

输出示例：

```json
{
  "items": [
    {"id": "note_1", "cursor": "..." },
    {"id": "note_2", "cursor": "..." }
  ],
  "next_cursor": "...",
  "complete": false
}
```

### D. Detail Fetcher

职责：

- 根据主键抓取详情页
- 优先解析页面状态
- 必要时使用浏览器兜底

### E. Extractor

职责：

- 从原始状态树中提取统一字段
- 输出内部标准模型

建议字段：

- `id`
- `title`
- `content`
- `author`
- `published_at`
- `metrics`
- `tags`
- `media`
- `source_url`

### F. Asset Downloader

职责：

- 下载图片 / 视频等资源
- 管理本地命名
- 去重保存

### G. State Store

职责：

- 保存进度
- 保存去重信息
- 保存失败信息
- 保存增量同步游标

### H. Scheduler / Rate Controller

职责：

- 控制并发
- 控制请求预算
- 批次冷却
- 统一重试
- 风控熔断

## 4.2 推荐数据流

```text
输入 -> 归一化 -> 会话准备
     -> 列表发现 -> 去重过滤 -> 详情抓取
     -> 字段抽取 -> 媒体下载 -> Markdown/JSON 落盘
     -> 更新进度状态 -> 进入下一批
```

## 5. 关于“访问频繁 / 安全访问报警”的正确处理方式

这一部分只讨论合规、通用、工程上有效的方法。

### 5.1 根因通常不在“请求不会发”，而在“请求预算失控”

高频报警常见原因：

- 同一任务重复扫全量
- 列表页和详情页重复请求过多
- 失败后立即重试，形成重试风暴
- 一个会话同时承担太多任务
- 没有本地缓存，导致同一内容被重复访问
- 评论、楼中楼、媒体下载全部默认开启，导致请求膨胀

先查预算，再谈代码。

### 5.2 默认策略应该是“少抓”

建议把默认抓取分级：

- Level A：只抓正文和基础元数据
- Level B：补图片
- Level C：补视频 / Live 图
- Level D：补评论
- Level E：补二级评论 / 转录等重任务

默认只开 A。  
只有明确需要时，再单独触发 B/C/D/E。

### 5.3 增量优先，不做全量重扫

这是最有效的降频方式。

实践要求：

- 维护 `seen_ids`
- 维护 `last_cursor`
- 维护 `last_success_at`
- 下一次只抓新增内容

如果没有新增，就不再进入详情抓取阶段。

### 5.4 为每类请求设预算

建议不要只做 `sleep`，而是做“预算控制”。

示例：

- 列表页：每轮最多 1 到 3 次
- 详情页：每轮最多 N 条
- 评论页：默认 0，按需开启
- 媒体下载：排到采集完成后单独执行

这样系统会自然地往“采集主信息优先”收敛。

### 5.5 使用随机等待，但不要迷信随机等待

随机等待有价值，但它只是辅助手段。

正确位置：

- 条目之间加小范围抖动
- 批次之间加较长冷却

错误理解：

- 只要随机 sleep 就能解决风控

不能。  
真正起作用的是：减少请求总量、减少重复访问、降低失败重试密度。

### 5.6 风控熔断

一旦检测到以下信号，直接进入冷却状态：

- 访问频繁
- 安全限制
- 登录异常
- 权限校验异常

系统动作应为：

1. 立即停止当前任务
2. 记录错误与上下文
3. 标记会话进入冷却窗口
4. 冷却结束前不再自动重试
5. 必要时要求人工确认

这是最重要的保护机制之一。

### 5.7 失败重试要分层

建议这样分：

- 网络抖动：可少量重试
- 临时超时：可少量重试
- 内容不存在：不重试
- 登录失效：转人工处理
- 风控命中：立即熔断，不自动重试

也就是说，重试系统不能只看“失败了”，要看“为什么失败”。

### 5.8 会话复用优于频繁新建

如果每条内容都新建浏览器上下文、刷新 cookie、重复打开页面，本身就会增加噪音。

建议：

- 一个批次复用一个登录会话
- 一个列表任务复用一个 page/context
- 任务结束再统一释放资源

这既减少系统开销，也减少访问面的不必要波动。

## 6. 给其他 Agent 的可复用原则

下面这些原则，建议直接作为通用开发规范。

### 原则 1：先找页面状态，再找接口

很多站点详情页都会把首屏数据注入 HTML。  
优先利用已有状态，而不是直接去追内部接口。

### 原则 2：先做增量，再做批量

没有增量机制的批量抓取，迟早会把请求预算打爆。

### 原则 3：先做幂等，再做并发

一条内容抓两次不应生成两份结果。  
去重和状态管理要先于提速。

### 原则 4：先抓元数据，再抓大资源

图片、视频、评论、转录都属于成本更高的任务。  
默认不要和元数据采集绑死。

### 原则 5：风控信号优先级高于业务完成率

命中风险限制时，系统首先要保护账号、会话和后续任务，不要为了一次任务完成率继续冲。

### 原则 6：把“人工接管”设计成正式能力

像登录失效、验证码、权限受限、风控冷却，这些都不应该被当成系统 bug。  
它们是正常运行边界的一部分，应该支持人工恢复。

## 7. 推荐落地方案

如果要继续演进当前实现，建议按下面顺序做。

### 第一阶段：把已有能力抽象干净

- 把列表抓取、详情抓取、媒体下载拆成独立模块
- 给错误分类建统一枚举
- 给进度状态建统一 schema

### 第二阶段：做正式的限速控制器

- 按任务类型配置预算
- 支持批次冷却
- 支持熔断和冷却窗口
- 支持按来源拆分限速策略

### 第三阶段：做缓存层

- 列表缓存
- 详情缓存
- 资源 URL 缓存
- 失败缓存

重点不是永久缓存，而是避免短时间重复访问同一目标。

### 第四阶段：做任务分级

- 轻量索引任务
- 完整归档任务
- 评论补抓任务
- 媒体补抓任务

这样才能把默认采集成本压下去。

## 8. 给开发同学的结论

当前 XHS 方案最核心的实现逻辑，可以概括成一句话：

> 用登录态和页面已有状态做低侵入采集，用本地状态和增量同步减少重复访问，用风控熔断保护任务边界。

真正通用、最值得迁移到其他 Agent 的，不是某个平台的字段细节，而是下面这套模式：

- 归一化输入
- 静态优先、浏览器兜底
- 列表与详情分离
- 本地状态驱动增量
- 元数据与大资源解耦
- 请求预算控制
- 风控熔断
- 人工接管

如果后续还要继续优化“访问频繁”的问题，优先级应该是：

1. 检查是否存在重复抓取
2. 检查是否默认开启了高成本能力
3. 检查是否缺少批次冷却和熔断
4. 检查是否缺少列表/详情缓存
5. 检查是否同一会话承担了过多任务

不要把问题理解成“还差一个绕过技巧”。  
大多数时候，问题是系统没有把采集过程当成一个有预算、有状态、有边界的同步任务来设计。
