# 输出模板

除非用户明确要求缩小范围，否则默认按下面的结构输出。

## 全局写作规则

每一个重要点都尽量采用下面的形式：

- 先给一句判断或总结
- 再在括号里补一段更具体的展开解释

括号里的展开解释通常应该回答三件事：

- 这句话具体是什么意思
- 有哪些论文内容、比较、现象或证据支持它
- 为什么这个点有 insight，或者为什么它对后续判断重要

如果某一节只是列了几个抽象标签，却不能让新手听懂，那一节就还没写够。

## 1. 领域前景

先用通俗中文解释：

- 这个领域想解决什么问题
- 为什么这个方向重要
- 当前前沿大致在往哪儿走
- 为什么这批论文能代表这个方向的一段演化

这一节要优先照顾初学者理解。
每个关键判断后都要补括号解释，而不是只给抽象总结。

## 2. 源论文锚点

明确指出哪篇是源论文。如果根目录下有一个和文件夹同名的 markdown 文件，默认把它视为源论文。

解释清楚源论文为什么是锚点。
通常可以覆盖它定义了什么问题、引入了什么关键机制、以及为什么后续论文会围着它展开，但不要求固定按这三句机械输出。
这些点都不应只写成短标签，后面要补括号解释其含义、证据和重要性。

## 3. 逐篇 Challenge 表

在画图之前，先给一张紧凑的 challenge 表或逐篇 bullet。

每篇重要论文至少包括：论文标题、它要解决的 challenge、对应场景、技术动作、以及留下的 technical insight。

这一节是强制的。后面的 idea 必须建立在这里，而不是凭空发散。
每条 challenge 分析都应带括号解释，而不是只剩名词短语。

## 4. 技术进步

总结这批论文真正推进了什么。

这一节只规定结构，不预设具体技术方向。你应该根据当前这批论文实际覆盖的主题，自行归纳出若干“真正发生推进”的技术维度。

写这一节时要做到：

- 归纳出这批论文实际推进了哪些方面
- 明确哪些推进是实质性的，哪些只是表述变化、系统堆叠或包装更新
- 尽量把“推进”落到机制、能力边界、评测覆盖或工程可行性上
- 不要为了套模板硬凑固定分类

对每个“推进点”，都补括号说明它具体推进了什么、证据来自哪里、为什么这不是表面改写。

## 5. 还没有解决的问题

列出读完整批论文后仍然存在的关键问题。

这一节同样只规定结构，不预设具体问题清单。你应该根据这批论文共同暴露出的局限，自行抽取最关键、最值得继续追问的未解决问题。

写这一节时要做到：

- 只保留真正反复出现、或者真正限制后续研究的关键问题
- 优先写“这批论文读完之后仍然没有被解决”的问题
- 尽量说明这些问题为什么难、为什么重要
- 不要为了套模板硬列几个常见 bottleneck

对每个问题，都补括号说明它在这批论文里如何暴露出来、为什么还没解决、为什么它会卡住后续工作。

## 6. 可复用 Insight

抽取这批论文里最值得继承的 technical insight。

每条 insight 通常应该交代这个模式是什么、哪些论文支持它、以及为什么它对后续工作重要；具体表述可根据语料调整。
不要把 insight 写成只有术语标签的短句，后面要有括号把它解释透。

## 7. follow-up 脉络图

然后给一段兼容 Obsidian 的 `[[Wiki Link]]` 链接图，并尽量以源论文作为整张图的根节点。

可参考的关系写法：

- `[[源论文]] -> [[后续论文A]] : 开出这条分支`
- `[[Paper A]] -> [[Paper B]] : 建立在其上`
- `[[Paper C]] -> [[Paper D]] : 属于其延伸`
- `[[Paper E]] -> [[Paper F]] : 同一瓶颈，不同机制`
- `[[Paper G]] <-> [[Paper H]] : 同一场景，不同技术路线`
- `[[Paper I]] <-> [[Paper J]] : 技术模块互补`

这一段应当作为主要输出形式，因为它可以直接粘贴进 Obsidian 笔记，并保留链接图谱。
不要输出“所有论文只连回源论文”的星型结构；只要有合理依据，就补上论文之间的交叉链接。

在人能读懂的脉络图后面，再补一份机器可读的关系清单，默认准备保存到 `$PWD/.paper_followup_links.md`。
这份清单里只保留标准关系行，每行一条，中间不要夹杂解释文字。
这份清单是 `scripts/update_obsidian_links.py` 的输入，用来把真实双链写回各论文 note。

推荐在生成关系清单后运行：

```bash
python3 scripts/update_obsidian_links.py \
  --root "$PWD" \
  --relations "$PWD/.paper_followup_links.md"
```

如果用户已经把关系行写进了 `Idea整理.md`，脚本也可以回退解析那里，但单独的隐藏关系文件更干净，也更适合重复更新。

如果需要再补一张 `mermaid` 图。

图后面还要解释这张图真正揭示了什么。
解释维度应从当前论文语料里长出来，而不是机械套固定提纲。
图后解释部分应当告诉读者“这些链接意味着什么”，并尽量配括号说明证据和 insight。

## 8. 论文分级

除非用户明确要求，否则不要把源论文和 follow-up 论文混在同一组排名里；先单独交代源论文，再评价 follow-up。

给 follow-up 论文选择一个符合这批论文信号结构的分组方式。
`高信号 / 中信号 / 低信号` 是默认分法，但不是强制。

每篇论文至少给出：

- 1 句话说它做了什么
- 1 句话解释为什么应该放在这个等级
- 如果有必要，再补 1 句话说学生应该从它身上学到什么

不要凭感觉分级，要基于证据。
每个判断后面都要补括号解释，说明支撑证据和为什么这个判断成立。

## 9. 二十个候选 idea

先给 20 个 idea，再做筛选。

每个 idea 至少包括：

- idea 标题
- 它在打什么瓶颈
- 它针对的具体场景是什么
- 核心技术假设是什么
- 它准备改什么 / 搭什么
- 它会用到什么模型、模块、目标、数据或信号
- 为什么这个时间点值得做
- 一个最小可验证实验是什么
- 它想达到什么效果

这 20 个 idea 要拉开差异，但不要强行套进固定分类。
它们应该从论文里提炼出的 insight、瓶颈、冲突和机会中自然采样出来。
可以同时有比较稳的 idea 和比较异想天开的 idea。
像“improve planning”这种没有机制细节的表述，不能算一个合格 idea。
每个 idea 的每个关键字段，最好都写成“主句 + 括号展开”的形式，让人能真正看懂这个 idea 想怎么做。

## 10. 第二轮筛选

重新回看这 20 个 idea，并给出明确的第二轮判断。
`靠谱 / 存疑 / 不靠谱` 是默认标签，但如果换一套边界更清晰的标签更合适，也可以换。

对每个 idea 解释：

- 为什么它留下了，或者为什么被淘汰
- 它最关键的隐性依赖或弱点是什么
- 它更像可发表方向、学习型项目，还是噪声方向

这一节里不要再“客气”。
每个筛选判断都要补括号说明依据，而不是只给一个标签。

## 11. 最推荐的方向

选出一小组最值得做的方向，通常是 1 到 3 个，但不必机械卡死。

对每个推荐方向，给出：

- 为什么它值得做
- 它应该建立在哪些 prior paper 上
- 第一组实验该怎么做
- 什么结果算成功
- 最大的失败模式是什么
- 即使失败，什么 fallback 路径还能留下有价值的积累

不要把这些只写成短词条。每一点后面都补括号解释机制、证据和实践意义。

## 12. 用户自带 Idea 评估

如果用户提供了自己的 idea，包括“靖文的这个 idea 怎么样”这类请求，单独加一节：

- 它在打什么 challenge
- 哪些论文支持它
- 哪些论文让它变得可疑
- 最直接的实现与验证路径是什么
- 给它一个清楚的判断；`靠谱 / 存疑 / 不靠谱` 可以作为默认标签，但不是唯一标签体系

这部分尤其不能简写。要在括号里解释为什么这个 idea 会得到这样的判断。

如果当前没有用户自带 idea，就明确写这一节暂缺输入。

## 13. Reading 与行动建议

最后用非常具体的话告诉新手：

- 先读什么
- 哪些论文只需要快读
- 先复现什么最合适
- 还缺什么背景或技能
- 现在先不要过度投入什么

每条建议都要补括号说明：为什么是这个建议、它和论文语料之间有什么关系。

## 写作规则

- 做重要判断时尽量点具体论文标题。
- 在讨论 follow-up 之前，先明确指出源论文是谁。
- 明确区分“论文声称”和“你的判断”。
- 避免空泛的 future work 套话。
- 风格上更像要求高的导师，不像宣传文案。
- 如果结论主要建立在 digest、摘要和引言上，要明确说明证据边界。
- idea 要具体到让学生能据此开一个第一版实验。
- 优先使用“主句 + 括号展开”的表达，而不是只给简短标签。
