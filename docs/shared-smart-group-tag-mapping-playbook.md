# 共享智能分组标签微调与 AI 整理规范

这份文档专门描述 `共享智能分组` 里的“标签微调”应该怎么做，目标是让人或 AI 助手都能按同一套规则整理 `原始标签 -> 共享规则 -> 共享组 -> 作者入组` 这条链路。

它不是产品宣传文档，而是一份可执行的操作规范。

## 适用范围

适用于下面几类工作：

- 给 `未确认` 原始标签补映射
- 修正明显错误的标签归组
- 让一个原始标签同时挂到多个共享组
- 评估是否应该新建共享组
- 在不重建整套标签库的情况下，只重排作者归组
- 给后续 AI 助手一套稳定、可复用的整理标准

不适用于下面几类工作：

- 修改平台卡片样式
- 直接手改作者名单而不通过标签归组
- 重写 XHS / B 站爬虫逻辑

## 核心概念

### 1. 原始标签

原始标签是从本地情报库内容里抽出来的信号，可能来自：

- frontmatter 中的 `tags` / `topics` / `keywords`
- Markdown 正文里的 `**标签**` / `**话题**` / `**关键词**`
- 小红书正文里的 `#话题`
- 专辑名 / 收藏夹名 / 文件夹名
- 标题中被提炼出来的紧凑关键词

例子：

- `Obsidian`
- `双链笔记`
- `卡片笔记`
- `联系导师`
- `26fall`
- `实验室vlog`

### 2. 共享规则

共享规则是 `原始标签 -> 共享组标签` 的映射关系。

例子：

- `Obsidian -> 知识管理 / Obsidian`
- `双链笔记 -> 知识管理 / Obsidian`
- `卡片笔记 -> 知识管理 / Obsidian`

现在系统也支持一个标签同时映射到多个共享组：

- `联系导师 -> 申博 / 留学 + 研究生 / 博士`

### 3. 共享组

共享组是最终给产品使用的稳定主题桶，用于：

- 按组筛选小红书博主
- 按组筛选 B 站 UP
- 按组驱动主动抓取
- 按组过滤情报 Feed

共享组应该是长期稳定、可读、可管理的主题标签，而不是临时碎词。

### 4. 作者入组

作者入组是最终结果。

系统会根据作者样本内容命中的原始标签，把作者加入一个或多个共享组。

例子：

- 某个小红书博主命中 `卡片笔记`
- 某个 B 站 UP 命中 `Obsidian`、`双链笔记`

这两个人都会被整理进：

- `知识管理 / Obsidian`

## 和 UI 状态的对应关系

### `未确认`

`未确认` 发生在标签层，意思是：

- 某个原始标签已经被扫描出来
- 但它还没有明确的共享组映射

这不等于作者一定没分组。

很多时候，作者可能已经靠其他标签进组了，只是这条标签本身还没有被正式编进共享规则词典。

### `待补标签`

`待补标签` 发生在作者层，意思是：

- 这个作者身上的可用标签和上下文都没有稳定命中共享组
- 系统只能先把他放到一个兜底桶里

`待补标签` 是要尽量缩小的，但不要求绝对为 0。

### `未分组`

`未分组` 是作者最终视图里的状态，常见原因有两种：

- 当前作者没有任何共享组
- 用户手动把这个作者从所有共享组里移掉了

### 共享智能分组相关文件

#### 1. 原始标签数据库

- `<vault>/.abo/shared-tag-database.json`

作用：

- 存文件级缓存
- 记录增量扫描状态
- 供后续只处理新增或变更笔记

#### 2. 共享标签摘要

- `<vault>/data/shared_tag_index.json`

作用：

- 给 UI 和人工查看用
- 汇总原始标签、出现次数、平台、样例标题、样例作者

#### 3. 共享组与规则

- `<vault>/data/shared_smart_groups.json`

作用：

- 保存共享组列表
- 保存 `signal_group_labels`
- 保存 `generated_at`

#### 4. 共享作者档案

- `<vault>/data/shared_creator_profiles.json`

作用：

- 保存 XHS / B 站作者最终归组
- 保存跨平台作者目录
- 保存作者样本标签和样本标题

## 单组映射与多组映射

### 单组映射

默认情况，一个原始标签应该只落到一个共享组。

适用场景：

- 语义稳定
- 下游只需要一个主要主题
- 不会误伤其他作者分组

例子：

- `Obsidian -> 知识管理 / Obsidian`
- `26fall -> 申博 / 留学`
- `护发精油 -> 变美 / 穿搭`

### 多组映射

只有在原始标签天然横跨两个稳定主题时，才允许多组映射。

建议约束：

- 默认最多 2 个共享组
- 只有在特别明确时才允许 3 个
- 不要把一个标签挂到 4 个以上的组

适用场景：

- 一个标签既表达“人生阶段”，又表达“主题方向”
- 一个标签对后续筛选确实有双重价值

例子：

- `联系导师 -> 申博 / 留学 + 研究生 / 博士`
- `实验室申请经验 -> 申博 / 留学 + 科研 / 学术`

不适用场景：

- 只是语义模糊
- 只是你不确定该归哪类
- 只是想让它“尽量都沾一点”

## 未确认标签应该怎么处理

### 基本原则

不要追求一次把所有未确认标签清零。

正确目标是：

- 优先处理高频、有影响力、会影响作者归组结果的未确认标签
- 保留少量低频、低价值、噪声型未确认标签

### 优先级分层

#### 第一优先级：必须尽快处理

满足任一条件就应该优先映射：

- 标签出现次数 `>= 3`
- 影响多个作者
- 横跨 XHS 和 B 站两个平台
- 当前有作者因为它掉进 `待补标签`
- 属于用户明显长期关注的方向

例子：

- `数字花园`
- `phd申请`
- `联系导师`
- `实验室vlog`

#### 第二优先级：建议处理

满足下面条件时建议补映射：

- 出现次数为 `2`
- 语义清晰
- 能明显并到已有共享组
- 对作者分组平衡有帮助

#### 第三优先级：可以暂时不处理

这些标签可以先留在未确认里：

- 只出现 1 次
- 很像标题碎片
- 活动词、临时词、时间词
- 纯情绪表达、口语残片、无稳定筛选价值

例子：

- `太绝了`
- `某次直播随手记`
- `今天有感`
- 某个一次性活动话题

### 四种处理动作

对每个未确认标签，最终只能做下面 4 种动作之一：

#### 1. 并入已有共享组

这是默认首选。

只要现有共享组已经能表达语义，就不要新建组。

例子：

- `数字花园 -> 知识管理 / Obsidian`
- `套磁 -> 申博 / 留学`

#### 2. 做成多组映射

只在有明确双重主题时使用。

例子：

- `联系导师 -> 申博 / 留学 + 研究生 / 博士`

#### 3. 新建共享组

只有在下面条件同时满足时才允许：

- 一批未确认标签明显聚成新主题
- 这个主题无法自然并入现有共享组
- 未来用户真的会用它做筛选
- 新组名能够长期稳定复用

#### 4. 暂时保留未确认

如果标签噪声大、出现少、上下文不稳定，就先不要强行映射。

## 共享组命名规范

共享组名必须满足下面规则：

- 简洁
- 稳定
- 可长期筛选
- 避免平台词
- 避免作者词
- 避免占位词

推荐格式：

- `主题A / 主题B`
- `一级主题 / 二级主题`

推荐例子：

- `知识管理 / Obsidian`
- `AI / 大模型`
- `研究生 / 博士`
- `申博 / 留学`
- `健康 / 医学科普`

禁止例子：

- `其他`
- `未确认`
- `待补标签`
- `低信息标签`
- `B站内容`
- `小红书博主`
- `视频`
- `笔记`
- `作者`

## 手动微调后的推荐操作

### 情况 1：只是改了标签映射

只需要执行：

- `仅整理博主 / UP`

用途：

- 复用已有共享标签库和共享组
- 只重排作者归组
- 更快

### 情况 2：最近新增了很多本地内容

执行：

- `共享智能分组`

用途：

- 重扫标签库
- 增量更新新标签
- 必要时补抓 B 站关注作者最近内容
- 再统一重排作者

## AI 助手整理时必须遵守的顺序

如果未来让 AI 助手来做这件事，必须按下面顺序执行。

### Step 1. 先确定 vault 路径

优先取当前应用配置的 vault 路径；如果上下文里没有，再回退到：

- `~/Documents/Obsidian Vault`

助手在开始整理时，必须明确记录：

- 当前 vault 路径
- 当前日期时间
- 本次准备执行 `full` 还是 `creator-only`

### Step 2. 先读现有共享组，不要从 0 造新组

优先读取：

- `<vault>/data/shared_smart_groups.json`

目标：

- 复用已有共享组名
- 减少标签漂移
- 降低下游筛选变化

### Step 3. 再读未确认标签的上下文

再读取：

- `<vault>/data/shared_tag_index.json`
- `<vault>/data/shared_creator_profiles.json`
- 必要时读取 `<vault>/.abo/shared-tag-database.json`

重点看：

- `count`
- `platforms`
- `sample_authors`
- `sample_titles`
- 当前是否已经影响某些作者分组

### Step 4. 先判断能否并入已有共享组

默认优先级：

1. 并入已有组
2. 多组映射
3. 新建组
4. 暂时不处理

不要反过来做。

### Step 5. 保存规则后，只重排作者

如果只是补映射，不要立刻要求全量重建。

默认推荐：

- 先保存 `signal_group_labels`
- 再执行 `creator-only`

只有在出现大量新增笔记、新增标签时，才执行 `full`

## 给 AI 助手的输出格式要求

如果让 AI 助手给出标签整理建议，建议让它输出结构化 JSON，而不是自由文本。

推荐格式：

```json
{
  "timestamp_shanghai": "2026-04-27 18:30",
  "vault_path": "/Users/huanc/Documents/Obsidian Vault",
  "workflow_mode": "creator-only",
  "decisions": [
    {
      "signal": "数字花园",
      "groups": ["知识管理 / Obsidian"],
      "action": "reuse-existing",
      "reason": "和 Obsidian、知识库、双链笔记高度同义"
    },
    {
      "signal": "联系导师",
      "groups": ["申博 / 留学", "研究生 / 博士"],
      "action": "multi-group",
      "reason": "既表达申博流程，也表达研究生/博士阶段"
    }
  ],
  "new_groups": [],
  "deferred_signals": [
    {
      "signal": "今天太顶了",
      "reason": "低频且没有稳定筛选价值"
    }
  ]
}
```

## 给 AI 助手的提示词模板

下面这段可以直接给后续 AI 助手用。

```text
你现在在维护 ABO 的共享智能分组标签映射。

你的任务不是直接改作者名单，而是整理“原始标签 -> 共享规则 -> 共享组 -> 作者入组”这条链路。

请遵守下面规则：

1. 先读取现有共享组，不要从 0 开始造组。
2. 优先复用已有共享组名，减少标签漂移。

  "workflow_mode": "creator-only or full",
  "decisions": [
    {
      "signal": "原始标签",
      "groups": ["共享组A", "共享组B"],
      "action": "reuse-existing | multi-group | new-group | defer",

    }

  "new_groups": [],
```

## 手动保存时的 JSON 约定

应用内部保存的 `signal_group_labels` 现在兼容下面两种形态：

### 单组映射

```json
{
  "Obsidian": ["知识管理 / Obsidian"]
}
```

### 多组映射

```json
{
  "联系导师": ["申博 / 留学", "研究生 / 博士"]
}
```

规范要求：

- 数组去重
- 顺序表示优先级，主组放前面
- 一般不要超过 2 个组
- 不要手改 `generated_at`、`updated_at`、`saved_at` 之类系统时间字段

## 时间记录规范

后续助手如果要产出整理报告，时间统一按下面规则写：

### 给人看的文档 / 报告

使用：

- `YYYY-MM-DD HH:mm Asia/Shanghai`

例子：

- `2026-04-27 18:30 Asia/Shanghai`

### 给程序的 JSON / 状态文件

使用系统生成的 ISO 8601 字段，不要人工重写：

- `updated_at`
- `generated_at`
- `saved_at`

这些字段由程序负责写入，助手不要手工伪造。

## 验收标准

整理完成后，至少要满足下面几点：

1. 高频未确认标签明显下降
2. `待补标签` 不能成为大桶
3. 同义标签尽量并到统一共享组
4. 同一主题在 XHS 和 B 站复用同名共享组
5. 多组映射只出现在确有必要的桥接标签上
6. 不出现泛词组名
7. 保存规则后执行 `creator-only`，作者归组结果能按预期刷新

## 推荐的人工复核清单

在把一批标签映射正式保存前，建议至少检查下面几项：

- 有没有把短词误吸到错误大类
- 有没有因为一时犹豫而滥用多组映射
- 有没有新建太多只用一次的共享组
- 有没有把平台词、活动词、口头词错误提升成共享组
- 这批调整是否真的会改善后续按组爬取和按组筛选

如果答案是否定的，就应该回到“优先复用已有组”的原则重新收敛。
