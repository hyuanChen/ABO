# 分析标准

把本地论文文件夹转成研究判断时，使用下面这套标准。

## 0. 最低解释标准

一个判断如果只有标签，没有展开解释，就还不算写完。

每个重要判断都应让读者看明白三件事：

- 这句话具体是什么意思
- 它的证据来自这批论文中的什么内容
- 为什么这个点有 insight，或者为什么它会影响后续研究判断

如果一个新手看完还不能把这个点复述出来，就说明解释仍然太压缩。

## 1. 这批论文的地图应该呈现什么

一张真正有用的 follow-up 图，不应该只是按时间列论文。它应该呈现：

- 这批论文里哪篇是源论文 / anchor paper
- 每篇论文在打什么瓶颈
- 这个瓶颈对应的具体场景是什么
- 每篇论文引入了什么核心机制
- 哪些 technical insight 值得后续继续复用
- 它为了解决问题牺牲了什么，或者做了什么取舍
- 它是 foundational、incremental，还是旁支
- 哪些瓶颈在不同分支里反复出现

如果根目录下存在一个与文件夹同名的 markdown 文件，默认把它视为源论文，并以它为参照去组织其他论文。

聚类轴应当主要从当前论文语料里长出来，而不是硬套预设分类。
如果需要举例，可以从机制、场景、评测方式、数据来源、系统角色、失败模式等角度归纳，但这些都只是参考，不是必须套用的固定桶。

## 2. 如何判断论文质量

当用户问哪些论文“好”、哪些论文“差”时，本质上是在问研究信号强弱。
不要停在标签本身；一定要把为什么得到这个判断讲清楚。

### 高信号论文

通常至少满足下面两条以上：

- 问题定义清楚，而且真的是领域瓶颈
- 提出的机制改变了模型能力，而不只是堆大模型
- 证据能支持“为什么有效”，而不只是结果更高
- 评测设置和论文声称的能力基本匹配
- 抽象层次好，能给后续工作提供可复用支点

### 中信号论文

这类论文通常仍然有用，但往往是：

- 在已有范式上做延伸
- 优化了某个子模块，但没有真正改变能力边界
- 有收益，但证据面偏窄
- 更适合当作工程参考或 baseline 参考

### 低信号论文

这类论文往往有一条或多条问题：

- 新意主要是重组已有套路或包装措辞
- 号称在解决瓶颈，但瓶颈定义不够清楚
- 证据强度匹配不上它的野心
- ablation 太弱，没法解释到底为什么有效
- 真正的收益来自规模、数据、基础模型，而不是它自己的贡献

这三档是一个好用的默认分法，但不是强制 taxonomy。
如果这批论文更适合用别的分组方式，就直接换，并说明理由。

必要时直接说清楚：

- “这是篇有工程参考价值的论文，但研究杠杆不强”
- “framing 有意思，但机制说服力不足”
- “benchmark 提升是真的，但 insight 密度偏低”

## 3. 如何判断研究 idea

每个 idea 至少要过下面几关：

### A. 它是不是在打一个真实瓶颈

- 这个问题是不是在论文语料里反复出现
- 更强的论文是不是也还没解决它

### B. 它有没有机制，而不只是口号

- 你准备改模型、目标、表征、训练流程中的哪一块
- 为什么这个改动能对准那个瓶颈

### C. 它对当前操作者是否可行

- 数据可得性
- 算力要求
- 机器人 / 硬件门槛
- 复现难度
- 评测是否清楚

### D. 它能不能产出可发表的信号

- 能不能形成清楚的 ablation 故事
- 能不能展示能力提升或理解提升
- 能不能在强 baseline 面前站住

### E. 它有没有长期延展空间

- 能不能长成 thesis 方向或平台方向
- 还是只是一次性的 patch

### F. 它是否具体到能开工

- 有没有说清楚到底改什么
- 有没有说清楚用什么模型、模块、loss、信号或数据
- 有没有给出最小实验
- 有没有说清楚期望达到什么效果

每个通过或不通过的判断，都要解释背后的因果和证据，不要只打分。

## 4. idea 标签

给完 20 个 idea 之后，第二轮必须给每个 idea 一个明确判断。
`靠谱 / 存疑 / 不靠谱` 是默认标签，但如果这批论文更适合别的标签体系，可以换，只要边界足够清楚。
不要让较弱的 idea 含糊过去。

## 5. 新手真正需要的帮助

初学者通常需要几层帮助。
常见的层次包括领域常识、源论文定位、challenge 地图、概念地图和可执行的第一步，但具体强调什么应由当前论文语料和用户需求决定。

所以最后的回答不能只停在判断上，还要明确说：

- 先读什么
- 先复现什么
- 什么事情不要先砸两周
- 什么样的第一步结果算是有效进展

这些建议都应该附带解释，不要只列条目。

## 6. 图谱质量要求

一个有用的 Obsidian 图谱，不能只是“所有论文都指向源论文”的星型结构。

更好的图谱应该把真正重要的关系显出来。
通常这会包括源论文到 follow-up 的关系、follow-up 之间的关系、分歧、替代机制、共享瓶颈、互补模块和跨分支联系，但不必把这些当成固定 checklist 来硬套。

## 7. 保持健康的怀疑

持续防止这些常见误判：

- 只看摘要就过度乐观
- benchmark 过拟合被误当成能力进步
- 标题听起来很大，机制其实很薄
- 真正贡献被 frontier model 的能力掩盖
- 真正起作用的是新数据或新基础设施，而不是论文提出的方法
- idea 听起来顺，但并没有具体到能搭建实验

如果本地语料不足以支持强结论，就明确降低置信度。
降低置信度时，也要解释为什么证据不够、缺的是什么。
